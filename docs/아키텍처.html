3. 게시판 → 채팅 → 친구 전환 시퀀스

서비스 KPI 직결되는 핵심 전환 플로우.

sequenceDiagram
  autonumber
  actor A as 사용자 A
  actor B as 사용자 B
  participant App as RN App
  participant API as Supabase API
  participant RT as Realtime
  participant Edge as Edge Fn

  A->>App: 게시글 작성
  App->>API: insert posts
  API->>Edge: trigger compute-points (+10P)
  Edge->>API: update points_log

  B->>App: 게시판 진입
  App->>API: select posts (RLS 적용)
  App-->>B: 리스트 표시

  B->>App: 댓글 작성
  App->>API: insert comments
  API->>RT: broadcast comment
  RT-->>A: 실시간 알림

  A->>App: B 프로필 → 채팅하기
  App->>API: insert chats + chat_members
  API->>RT: subscribe channel

  loop 메시지 교환
    A->>App: 메시지 전송
    App->>API: insert messages
    API->>RT: broadcast
    RT-->>B: 메시지 수신
  end

  A->>App: 친구 추가
  App->>API: insert friends(pending)
  B->>App: 알림 → 수락
  App->>API: update friends(accepted)
  Note over A,B: 마이룸 방문 + 재대화 채널 형성

4. 배포 파이프라인 (CI/CD)

GitHub push → Actions → 각 대상 자동 배포.

flowchart TB

  SBCli --> Functions[Edge Functions]

  Actions --> Tests[Lint + Type + Tests]

  VercelL --> Browser2[(방문자 브라우저)]

5. 보안 / 권한 레이어

RLS 중심 권한 모델, Service Role 은 서버 전용.

flowchart TB
  subgraph Client[클라이언트]
    Anon[anon key]
    Authed[로그인 JWT]
  end
  subgraph Server[서버 전용]
    SR[Service Role Key]
  end
  subgraph SB[Supabase]
    AuthSvc[Auth]
    PostgREST
    RLS{RLS Policy}
    Tables[(Tables)]
  end

  Anon --> AuthSvc
  AuthSvc -->|로그인| Authed
  Authed --> PostgREST
  Anon --> PostgREST
  PostgREST --> RLS
  RLS --> Tables
  SR -. RLS 우회 .-> Tables

  subgraph Policy[정책 매트릭스]
    P1[posts: SELECT public, INSERT auth, UPDATE/DELETE 본인]
    P2[messages: chat 멤버만]
    P3[friends: 관계자만]
    P4[reports: INSERT auth, SELECT 관리자]
    P5[waitlist: INSERT anon, SELECT 관리자]
    P6[profiles: SELECT public, UPDATE 본인]
  end
  RLS --- Policy

flowchart LR
