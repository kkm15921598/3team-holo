# HOLO — 일반 사용자 Supabase Auth 전환 계획서

작성: 2026-06-17

---

## 1. 왜 / 무엇을

**지금 문제**
일반 사용자는 직접 만든 `users` 테이블 + 공개(anon) 키로만 동작한다. DB는 요청자가 누구인지 모른다(`auth.uid()` 없음). 그래서 "본인 글만 수정/삭제" 같은 **사용자별 권한을 걸 수 없다.** (`posts.user_id`가 전부 NULL인 것도 이 때문)

**목표**
일반 사용자를 **Supabase Auth(슈퍼베이스 내장 로그인 시스템)** 로 옮겨, 로그인하면 DB가 `auth.uid()`로 사용자를 식별하게 한다. 그러면 진짜 행 단위 보안(RLS)이 가능해진다.

**지금이 적기인 이유**
- 실사용자가 없다(냥님 테스트 계정뿐) → **기존 회원 이전 작업이 통째로 불필요**, 제일 위험한 부분이 사라짐.
- 로그인이 깨져도 테스트 계정만 영향 → 자유롭게 실험 가능.
- 출시 전이 Auth를 제대로 까는 최적 타이밍.

**외부 연동 필요 없음**
Supabase Auth는 슈퍼베이스에 내장. 별도 프로그램·SMS 업체 불필요(전화번호를 합성 이메일로 변환해 사용).

---

## 2. 핵심 설계 결정

1. **로그인 방식 유지(전화번호 + 비밀번호)**
   Supabase Auth는 "이메일 + 비밀번호"를 쓴다. 앱은 전화번호로 로그인하므로, 전화번호를 **합성 이메일**로 바꿔 Auth에 등록한다.
   예) `01012345678` → `01012345678@holo.local`
   (실제 메일 발송 없음 — 관리자 `admin@holo.app`과 같은 방식)

2. **매핑으로 코드 최소 수정** ⭐
   `users` 테이블에 **`auth_id`(uuid)** 컬럼을 추가해 Auth 계정과 연결한다.
   RLS 정책은 이 매핑을 이용해 **기존 `user_phone` / `nickname` 컬럼의 소유권**을 판정한다.
   → 글 작성자를 닉네임으로 쓰는 **1137곳 코드를 안 건드려도 된다.** (이게 작업량을 크게 줄이는 핵심)

---

## 3. 현재 코드 규모 (실측)

| 항목 | 규모 | 이번 작업에서 |
|---|---|---|
| `getCurrentAccount`(로그인 상태=전화번호) | **39개 파일** | 세션 기반으로 전환 (래퍼 1개로 최소화) |
| `authorNickname`(작성자 식별) | **1137곳** | **안 건드림** (매핑으로 해결) |
| 로그인/가입/비번 핵심 호출 | 약 5곳 | Auth 방식으로 교체 |

## 4. 단계별 진행 (각 단계 후 배포 → 테스트 → 다음)

> ⚠️ 제가 여기서 빌드·테스트를 못 하므로, 공지 고칠 때처럼 **"코드 수정 → 냥님 배포 → 테스트 → 안 되면 수정"** 을 반복합니다. 한 번에 안 끝납니다.

### 0단계 — 준비 (위험 없음)
- `users`에 `auth_id uuid` 컬럼 추가 (SQL, nullable).
- Auth Email 제공자 켜짐 확인(이미 됨), 회원가입 토글은 전환 중 임시로 **다시 켜야 함**(가입 테스트 위해).
- 기존 테스트 계정 목록 정리.

### 1단계 — 회원가입을 Auth로 (코드)
- 가입 시: `users` 직접 insert 대신 **먼저 `supabase.auth.signUp({ email: 합성이메일, password })`** → 성공하면 `users` 행 생성 + `auth_id` 저장.
- 영향: 가입 흐름만. 테스트 계정으로 확인.

### 2단계 — 로그인을 Auth로 (코드)
- 로그인: `verify_login` RPC → **`supabase.auth.signInWithPassword({ email: 합성이메일, password })`**.
- "현재 로그인 사용자"를 Auth 세션에서 가져오게 변경.
- 영향: **로그인이 깨지면 못 들어감** → 테스트 계정으로 단독 검증, 실패 시 즉시 코드 롤백.

### 3단계 — 세션·식별 정리 (코드)
- `getCurrentAccount`(39곳)가 Auth 세션 기준으로 전화번호를 얻도록 **래퍼 함수 한 곳만** 바꿔 파급 최소화.
- 비밀번호 변경/찾기도 Auth 방식으로.

### 4단계 — RLS 정책 적용 (SQL) ← 진짜 권한
- `users`: 본인 행만 수정 (`auth_id = auth.uid()`).
- `posts`/`comments`/`messages` 등: `users` 매핑으로 소유권 판정 → 본인 것만 수정/삭제.
- 한 테이블씩 단계적으로(깨지면 그 테이블만 롤백).

### 5단계 — 마무리 검증
- 익명(anon) 쓰기 범위 축소, 전체 흐름(가입→로그인→글쓰기→수정/삭제→채팅) 점검.
- 회원가입 토글 정책 최종 결정.

---

## 5. 롤백 전략

- **코드**: 각 단계 = 별도 커밋 → 문제 시 `git revert` + 재배포.
- **DB**: 대부분 "정책 추가"라 rollback SQL로 정책만 제거하면 원복.
- **계정**: 실사용자 없으니 테스트 계정을 언제든 초기화(삭제/재생성) 가능.

---

## 6. 위험 / 주의

- 이 작업은 **로그인 자체를 교체**한다. 단계 2가 제일 민감(로그인 깨짐 가능) → 반드시 단독 테스트.
- 빌드 테스트 불가 환경 → **배포 후 실제 확인**이 필수. 콘솔(F12) 에러를 같이 보면 빠르다.
- 반드시 **한 단계씩**. 한 단계 확인되기 전 다음으로 안 넘어감.

---

## 7. 예상 규모

- **1~3단계(코드)** = 작업의 대부분. 여러 번의 수정·배포·테스트 반복 예상.
- **4단계(SQL)** = 상대적으로 작음(여기가 그동안 말한 "SQL").
- 하루에 안 끝나며, 며칠에 걸쳐 단계별로 진행하는 프로젝트.
