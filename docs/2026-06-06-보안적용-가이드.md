# HOLO 보안 적용 가이드 (Supabase Auth + RLS)

작성: 2026-06-06

Security Advisor의 "RLS Disabled in Public" 에러 9건을 해결하기 위해 자체 로그인(평문 비밀번호 비교)을 Supabase Auth 로 전환하고, 모든 테이블에 RLS(행 수준 보안)를 적용했다.

## 적용 순서 (반드시 이 순서대로)

1. **대시보드 설정 변경** — Supabase 대시보드 → Authentication → Sign In / Providers → Email → **"Confirm email" OFF**. 휴대폰 번호를 가상 이메일(`01012345678@holo-user.app`)로 변환해 가입하므로 실제 메일 확인이 불가능하다. 이걸 안 끄면 가입·로그인이 전부 막힌다.
2. **SQL 실행** — SQL Editor 에서 `docs/sql/2026-06-06-auth-rls.sql` 전체를 붙여넣고 RUN. 여러 번 실행해도 안전하다.
3. **새 코드 배포** — 이 커밋의 클라이언트 코드를 배포(Vercel 등). SQL 실행 시점부터 구버전 배포본의 로그인·가입은 동작하지 않으므로 2번과 3번은 같은 시점에 진행할 것.
4. **확인** — Advisors → Security Advisor → Rerun linter. "RLS Disabled in Public" 에러가 사라져야 한다.

## 무엇이 어떻게 바뀌었나

서버(SQL): users 에 `auth_id` 컬럼 추가(Auth 계정 연결), 8개 테이블 + 부가 테이블 RLS 활성화, 로그인 전 화면용 RPC 함수 8개 추가(중복확인·아이디/비번 찾기·계정이전·탈퇴).

클라이언트: 로그인/가입이 `supabase.auth` 를 쓰도록 전환. 수정 파일은 `shared/lib/auth.ts`(신규), login-screen, review-screen, verification-screen, nickname-screen, find-id-screen, find-password-screen, password-change-screen, mypage-screen(로그아웃), account-screen(탈퇴).

## 기존 회원은 어떻게 되나

기존 회원은 **첫 로그인 때 자동으로 Auth 계정으로 이전**된다(같은 번호·비밀번호 그대로). 이전되는 순간 서버에 저장돼 있던 평문 비밀번호는 NULL 로 지워진다. 배포 직후 모든 사용자는 한 번 다시 로그인해야 한다(기존 세션 개념이 없었으므로 로그인 화면으로 보내짐).

## 보안 수준 요약

이제 보장되는 것: 로그인 없이는 어떤 데이터도 읽기/쓰기 불가. 메시지·채팅방은 방 멤버만, 알림·친구목록은 본인만 접근 가능. 글·댓글은 본인 명의로만 작성, 비밀번호는 해시로만 저장.

남은 한계(앱 구조상 의도된 트레이드오프): 게시글 update 는 로그인 사용자 전체에 허용(좋아요수·참여자 갱신을 다른 사용자가 하기 때문 — 더 조이려면 카운터를 트리거/RPC 로 분리). 로그인한 사용자끼리는 users 테이블의 프로필 정보(닉네임·레벨 등과 phone)를 조회할 수 있다(앱 전체가 phone 을 식별자로 쓰는 구조라 숨기려면 대규모 리팩토링 필요).

## 전원 이전이 끝난 뒤 (선택)

모든 기존 회원이 한 번씩 로그인한 뒤 SQL Editor 에서:

```sql
update public.users set password = null where auth_id is not null;
-- 더 지나서: alter table public.users drop column password;
```
